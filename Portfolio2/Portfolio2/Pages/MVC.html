MVC

The MVC activities and project were the group's first introduction to true Full Stack development. We were given an already set-up project/solution that was ready to be used for the MVC Framework. We then had to go through a process of creating the service, controller, and views. This taught us a lot of new information as well as gave us some important tips to remember.

For this project we all split off separately each working on our own view. This allowed for each of us to get a feel for the Full Stack development. This was not too challenging as our tass was very simple, complete unit conversions on the server side. We simply received data from a form in the view, brought it to the controller to call a method in the service which passed the information in form and converted the units. This was then sent back to the new view with the conversion passed to it using razor code. This allowed us to do the calculations, which was not very difficult.

The real difficulty for me came from the styling and user experience. In our previous JavaScript unit conversion project we had a simple styling where you could enter a number into one of the two input boxes and the opposing one would change with the converting unit allowing for the user to convert in either direction. We wanted to recreate that in the MVC unit conversion project which posed more challenges than expected. Our first change was that we were using a form submission rather than JavaScript, this meant that we needed a submit button. Our solution to this was to change our equivalent symbol in between the two input boxes to be our form submission button. This didn’t cause too many issues, just a simple change from a header to a submit button. The next biggest issue was choosing our input. With our design we had two input boxes which could both take input at the same time, this causes an issue for the converting as it could just convert both and place it into the opposing box in the new view. However, then the user would not be able to see their initial input so the information would not make any sense. To fix this I looked at JavaScript to not allow for the user to input into both boxes. When the user is typing into one input box it would set the other equal to zero. Simply, in the service we checked for which input was not equal to zero and converted that one. This solved our problem and allowed us to keep the sleek interface from the previous project.
